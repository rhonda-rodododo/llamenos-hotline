# Contact Import/Export & Merging — Design Spec

**Goal:** Enable admin-level CSV import with dedup, encrypted export for backup/portability, and a merge UI for resolving duplicate contacts.

**Depends on:** Spec 0 (PBAC Redesign), Spec 1 (Tag Management), Contact Directory v1 (PR #26).

---

## 1. CSV Import

### Use case

A hub admin has a spreadsheet of contacts from a partner org, a previous system, or a rapid response event. They need to bulk-import these into the contact directory with proper encryption and dedup.

### Import flow

1. Admin navigates to `/contacts/import`
2. Uploads a CSV file (client-side parsing — file never sent to server as plaintext)
3. Client shows a column mapping UI: map CSV columns to contact fields
4. Client previews first 10 rows with mapped fields
5. **Dedup check**: for each row with a phone number, client calls `POST /api/contacts/hash-phone` to get the HMAC, then `GET /api/contacts/check-duplicate` to check for existing contacts
6. Preview shows: N new contacts, M potential duplicates
7. For duplicates: "Skip" / "Import Anyway" / "Merge" per row (or bulk choice)
8. Client encrypts each contact (Tier 1 + Tier 2 envelopes) and sends to `POST /api/contacts/import`
9. Progress bar for large imports
10. Summary: N created, M skipped, K errors

### Column mapping

Standard columns:
- Display Name (→ Tier 1 encryptedDisplayName)
- Full Name (→ Tier 2 encryptedFullName)
- Phone (→ Tier 2 encryptedPhone + identifierHash)
- Email (→ Tier 2 PII blob)
- Type (→ plaintext contactType)
- Risk Level (→ plaintext riskLevel)
- Tags (→ plaintext tags, comma-separated)
- Notes (→ Tier 1 encryptedNotes)

Custom fields: mapped by custom field name/label.

### Batch API

```
POST /api/contacts/import
```

```typescript
{
  contacts: Array<{
    contactType: string
    riskLevel: string
    tags: string[]
    identifierHash?: string
    // All encrypted fields same as POST /api/contacts
    encryptedDisplayName: string
    displayNameEnvelopes: RecipientEnvelope[]
    // ... Tier 2 fields
  }>
}
```

Returns: `{ created: number, errors: Array<{ index: number, error: string }> }`

Requires: `contacts:create` + `contacts:envelope-full`

### E2EE: client-side encryption

The CSV is parsed and encrypted entirely in the browser. The server receives only encrypted blobs + plaintext metadata (type, risk, tags). This preserves zero-knowledge — the server never sees imported PII.

### Rate limiting

Import is limited to 500 contacts per batch to prevent abuse and manage envelope encryption cost (each contact needs ECIES envelopes for all authorized recipients).

## 2. Encrypted Export

### Use case

Admin needs a backup, or needs to share contact data with a partner org, or is migrating to a different system.

### Export flow

1. Admin navigates to `/contacts/export` or clicks "Export" in the directory toolbar
2. Chooses export scope: "All contacts" or "Selected contacts" (if multi-select active)
3. Chooses format: CSV (plaintext, for external use) or Encrypted JSON (for backup/migration)
4. **CSV export**: client decrypts all contact fields, generates CSV in browser, triggers download. Server not involved — purely client-side.
5. **Encrypted JSON export**: client generates a JSON file with all encrypted blobs + metadata, suitable for re-import into another Llámenos instance.

### CSV export (plaintext)

Requires: `contacts:read-all` + `contacts:envelope-full` (must be able to decrypt everything)

The CSV is generated in the browser after client-side decryption. The file is never sent to or generated by the server. Contains:
- Display Name, Full Name, Phone, Email, Address, DOB
- Type, Risk Level, Tags
- Custom field values
- Support contact names and relationships (denormalized)

### Encrypted JSON export (backup)

```typescript
interface ContactExport {
  version: 1
  hubId: string
  exportedAt: string
  exportedBy: string  // pubkey
  contacts: Array<{
    // All fields from the contacts table, as-is (encrypted)
    id: string
    contactType: string
    riskLevel: string
    tags: string[]
    encryptedDisplayName: string
    displayNameEnvelopes: RecipientEnvelope[]
    // ... all encrypted fields
  }>
  relationships: Array<{
    encryptedPayload: string
    payloadEnvelopes: RecipientEnvelope[]
  }>
}
```

This can be re-imported into another instance (envelopes would need re-wrapping for the new instance's users).

## 3. Contact Merging

### Use case

Two contacts exist for the same person (created by different volunteers, or imported with slight variations). An admin or case manager needs to merge them.

### Merge flow

1. From a contact's profile, click "Merge with another contact"
2. Search/select the other contact to merge with
3. **Side-by-side comparison**: shows both contacts' decrypted fields
4. For each field, user picks which value to keep (or edits a merged value)
5. Preview: merged contact + what happens to linked data
6. Confirm → merge executes

### Merge logic

1. Choose a "primary" contact (the one that survives)
2. All linked calls, conversations, notes, reports, and relationships from the secondary contact are re-linked to the primary
3. Tags are unioned
4. The secondary contact is soft-deleted (keeps a `mergedInto: primaryId` reference for audit)
5. The primary contact's fields are updated with the merge choices

### API

```
POST /api/contacts/:primaryId/merge
```

```typescript
{
  secondaryId: string
  // Merged field values (re-encrypted by client)
  encryptedDisplayName: string
  displayNameEnvelopes: RecipientEnvelope[]
  // ... all fields that changed
  contactType: string
  riskLevel: string
  tags: string[]
}
```

Server:
1. Re-links all `contact_call_links` from secondary → primary
2. Re-links all `contact_conversation_links` from secondary → primary
3. Updates notes with `contactHash` matching secondary's `identifierHash` → primary's hash
4. Re-links relationships (decrypt-re-encrypt is client-side; server just stores new blobs)
5. Updates primary contact with merged fields
6. Soft-deletes secondary with `mergedInto` reference
7. Audit log entry

Requires: `contacts:update-all` + `contacts:envelope-full` + `contacts:delete`

### Schema addition

Add to contacts table:
```typescript
mergedInto: text('merged_into')  // ID of the contact this was merged into (null if not merged)
```

## 4. Duplicate Detection Improvements

Beyond phone HMAC matching (current):

### Suggestions on contact list

When viewing the contact directory, show a "Possible duplicates" alert if the system detects contacts with matching `identifierHash` values. This is server-side (HMAC comparison) and doesn't require decryption.

### Future: fuzzy matching

Not in this spec, but the architecture supports it:
- Client-side: after decrypting display names, detect similar names (Levenshtein distance)
- Show "Did you mean [existing contact]?" during creation

## 5. Permissions

| Action | Permission |
|--------|-----------|
| Import contacts | `contacts:create` + `contacts:envelope-full` |
| Export CSV (plaintext) | `contacts:read-all` + `contacts:envelope-full` |
| Export encrypted JSON | `contacts:read-all` |
| Merge contacts | `contacts:update-all` + `contacts:envelope-full` + `contacts:delete` |
| View duplicate suggestions | `contacts:read-all` |

---

## 6. Non-Goals

- Import from other CRM systems (future — would need per-system adapters)
- Automatic merge without human review
- Real-time duplicate detection during import (dedup check is batch, not streaming)
- Cross-hub import/export (each hub is independent)
- Encrypted export re-import with automatic envelope re-wrapping (future — requires key exchange between instances)
