# Ops Infrastructure — Design Spec

**Date:** 2026-03-22
**Status:** Draft

## Scope

Three related operational gaps:

1. **PostgreSQL backup** — no automated backup exists; data loss risk on host failure.
2. **MinIO initialisation** — bucket is never auto-created; first file upload will fail on a fresh deploy.
3. **Systemd service** — no systemd unit exists; Docker Compose does not start after host reboot.

These are grouped because they all live in `deploy/` and are applied together by Ansible.

---

## Part 1: PostgreSQL Backup

### Requirements

- Automated daily backup of the PostgreSQL database
- Backups are encrypted at rest before leaving the host (client-side encryption)
- GFS retention: 7 daily, 4 weekly, 3 monthly backups kept
- Backups stored off-host (S3-compatible: AWS S3, Backblaze B2, or Hetzner Object Storage)
- Test-restore procedure documented and runnable
- Health endpoint includes backup freshness status

### Design

**Backup script** — generated by the Ansible backup role as an inline template, not as a standalone file in `deploy/scripts/`. Modify the backup script content directly in the Ansible backup role (`deploy/ansible/roles/backup/tasks/main.yml`). Do not create a separate `deploy/scripts/backup-postgres.sh` file. The script must implement:

```
1. pg_dump → compressed .sql.gz in /tmp
2. age-encrypt with operator's SSH public key (age: simple, widely-audited)
3. Upload to S3 with date-based key: backups/daily/YYYY-MM-DD.sql.gz.age
4. Apply GFS rotation: delete daily backups older than 7 days; keep weekly (Monday), monthly (1st)
5. Write timestamp to /var/lib/llamenos/last-backup-at (read by health endpoint)
```

`age` is chosen over GPG for simplicity — single command, no keyring management, clear failure modes. Backup can be decrypted with: `age --decrypt -i ~/.ssh/id_ed25519 backup.sql.gz.age | gunzip | psql`

**Environment variables required:**
- `BACKUP_S3_BUCKET` — target bucket
- `BACKUP_S3_ENDPOINT` — S3-compatible endpoint URL
- `BACKUP_S3_KEY_ID` + `BACKUP_S3_SECRET` — credentials
- `BACKUP_AGE_PUBLIC_KEY` — operator's age/SSH public key for encryption

**Systemd timer** (preferred over cron for reliable execution and logging):
```
backup-llamenos.timer  — daily at 03:00 UTC
backup-llamenos.service — one-shot, runs backup script
```

**Important:** The existing cron job in the backup Ansible role (`deploy/ansible/roles/backup/tasks/main.yml`) must be REMOVED and replaced with the systemd timer. Do not add the timer alongside the existing cron job — backups will run twice otherwise.

**Restore script** `deploy/scripts/restore-postgres.sh`:
- Takes backup file path as argument
- Decrypts with age, gunzips, runs psql restore
- Documents connection string requirements

### Health Endpoint Integration

`GET /api/health` adds:
```json
"backup": {
  "status": "ok" | "stale" | "unknown",
  "lastBackupAt": "2026-03-22T03:00:00Z" | null
}
```
Reads `/var/lib/llamenos/last-backup-at`. `stale` if >25 hours old (allows for some timer variance). `unknown` if file doesn't exist (fresh install or never run).

**Prerequisite:** The backup script (in the Ansible role) must write a success timestamp to `/var/lib/llamenos/last-backup-at` after each successful run. The health endpoint depends on this file. If the health endpoint is updated before the backup script writes the file, `/var/lib/llamenos/last-backup-at` may not exist — the health check must return `"unknown"` (not an error) when the file is absent.

## Part 2: MinIO Initialisation

### Problem

On a fresh deploy, the MinIO bucket does not exist. The application will fail on the first file upload with a `NoSuchBucket` error that is not surfaced clearly to operators.

### Design

**Init script** `deploy/scripts/init-minio.sh`:

```
1. Wait for MinIO to be ready (mc alias set with retry)
2. Create bucket if not exists (mc mb --ignore-existing)
3. Set lifecycle rules:
   - recordings/: expire after 90 days
   - voicemails/: expire after 365 days
4. Create dedicated IAM user (MINIO_APP_USER / MINIO_APP_PASSWORD)
5. Create and attach least-privilege policy (GetObject, PutObject, DeleteObject, ListBucket on bucket only)
```

The script is **idempotent** — safe to run on every deploy. Step 4-5 use `mc admin user add --allow-exists`.

**Why a dedicated IAM user?** The app currently uses root credentials. If the app is compromised, the attacker has full MinIO admin access. The dedicated user can only access the one bucket.

**Ansible integration:** Add task after container startup to copy and run `init-minio.sh`.

### Application Code

Update `src/platform/node/blob-storage.ts`:
- Use `MINIO_APP_USER`/`MINIO_APP_PASSWORD` if set; fall back to root for dev only
- Document in env config

### Health Endpoint

Add `minio` check to `GET /api/health`:
```json
"minio": { "status": "ok" | "bucket_missing" | "unreachable" }
```
Uses S3 `headBucket` call. `bucket_missing` means MinIO is up but init-minio.sh was never run.

## Part 3: Systemd Service

### Problem

No systemd unit manages the Docker Compose stack. On host reboot, services do not start automatically unless Docker daemon is configured with `--restart=always` *and* containers were started with `--restart=unless-stopped`. The current setup relies on Docker's own restart policy, which is fragile.

### Design

`deploy/systemd/llamenos.service`:

```ini
[Unit]
Description=Llamenos Hotline Stack
Requires=docker.service
After=docker.service network-online.target
Wants=network-online.target

[Service]
Type=oneshot
RemainAfterExit=yes
WorkingDirectory=/opt/llamenos
ExecStart=/usr/bin/docker compose -f /opt/llamenos/docker-compose.yml up -d --remove-orphans
ExecStop=/usr/bin/docker compose -f /opt/llamenos/docker-compose.yml down
TimeoutStartSec=120
TimeoutStopSec=60

[Install]
WantedBy=multi-user.target
```

**Ansible deploys the unit:**
1. Copy `llamenos.service.j2` template to `/etc/systemd/system/llamenos.service`
2. `systemctl daemon-reload`
3. `systemctl enable --now llamenos`

This ensures the stack starts on boot and can be managed with `systemctl start/stop/restart llamenos`.

---

## Deployment Coordination

All three pieces are applied in a single Ansible role task sequence:
1. Start containers (existing)
2. Run `init-minio.sh` (new, idempotent)
3. Install systemd unit + enable (new)
4. Install backup timer (new)
5. Verify health endpoint returns all components ok

---

## Testing

**Backup:**
- Run backup script → verify encrypted file appears in S3
- Run restore script → verify database contents match
- Set last-backup-at to 26h ago → health endpoint reports `stale`

**MinIO init:**
- Fresh deploy → `init-minio.sh` runs → bucket exists, app user exists
- Run again → idempotent (no errors)
- Health endpoint reports `minio: ok`

**Systemd:**
- `systemctl is-active llamenos` → active after Ansible run
- `systemctl is-enabled llamenos` → enabled
- Simulate reboot (restart Docker daemon) → stack recovers automatically
