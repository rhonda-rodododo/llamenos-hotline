# Spec 1: LiveKit Infrastructure & VideoAdapter

**Status:** Deferred
**Date:** 2026-04-03
**Dependencies:** None (foundation spec)
**Blocked by:** Nothing — can be implemented first

## Context

Llamenos is evolving from a crisis hotline into a generalized secure communications hub for activist organizations. Video conferencing is needed for volunteer trainings (webinar-style, occasionally hybrid with breakout rooms), team coordination, and admin briefings. LiveKit is selected as the self-hosted SFU — it aligns with the existing Docker/Ansible deployment, supports SFrame E2EE, and has TypeScript-first SDKs.

Meetings are a completely separate system from the hotline call flow. No integration between TelephonyAdapter and VideoAdapter.

## Deployment Topology

### New Docker Services

```yaml
livekit-server:
  image: livekit/livekit-server:latest
  ports:
    - "7880:7880"   # HTTP API + WebSocket signaling
    - "7881:7881"   # RTC (WebRTC UDP/TCP)
    - "50000-50100:50000-50100/udp"  # ICE UDP port range
  environment:
    - LIVEKIT_KEYS=${LIVEKIT_API_KEY}:${LIVEKIT_API_SECRET}
    - LIVEKIT_REDIS_ADDRESS=redis:6379
  depends_on:
    - redis

redis:
  image: redis:7-alpine
  ports:
    - "6379:6379"
  # LiveKit requires Redis for room state coordination
  # Can be shared with other services if needed later

livekit-egress:
  image: livekit/egress:latest
  environment:
    - EGRESS_CONFIG_BODY=<deployment-specific>  # Points to LiveKit server + Redis; generated by Ansible
  # Egress handles server-side recording compositing
  # See Spec 3 for recording pipeline details
  profiles:
    - recording  # Optional — only needed if recording is enabled
```

### TURN Reuse

The existing coturn instance serves both voice (SIP WebRTC) and video (LiveKit). LiveKit is configured with external TURN credentials via its server config:

```yaml
turn:
  enabled: true
  domain: turn.example.com
  tls_port: 5349
  udp_port: 3478
  external_tls: true
```

coturn's time-limited HMAC credentials work with LiveKit's TURN integration. No changes to coturn itself.

### Caddy Integration

Caddy adds a reverse proxy rule for LiveKit's WebSocket endpoint:

```

}
```

## VideoAdapter Interface

Located at `src/server/video/adapter.ts`. Follows the existing adapter pattern (TelephonyAdapter, MessagingAdapter) but simpler — single implementation (LiveKit) with the interface allowing future swaps.

### Interface Definition

```typescript
export interface VideoRoom {
  roomId: string;
  hubId: string;
  name: string;
  numParticipants: number;
  maxParticipants: number;
  createdAt: Date;
  e2eeEnabled: boolean;
}

export interface VideoParticipant {
  identity: string;        // pubkey or contactId
  name: string;            // display name (encrypted in meeting record)
  role: 'host' | 'presenter' | 'participant' | 'guest';
  joinedAt: Date;
  isPublishing: boolean;   // has active audio/video tracks
  isMuted: boolean;
}

export interface VideoToken {
  token: string;           // LiveKit JWT
  url: string;             // WebSocket URL to connect
  roomName: string;
}

export interface RecordingOptions {
  layout: 'speaker' | 'grid';
  outputFormat: 'mp4' | 'webm';
  audioOnly: boolean;
}

export abstract class VideoAdapter {
  abstract createRoom(hubId: string, options: {
    name: string;
    maxParticipants: number;
    e2eeRequired: boolean;
    emptyTimeout: number;    // seconds before empty room auto-closes
  }): Promise<VideoRoom>;

  abstract deleteRoom(hubId: string, roomId: string): Promise<void>;

  abstract generateToken(hubId: string, roomId: string, identity: string, options: {
    role: 'host' | 'presenter' | 'participant' | 'guest';
    displayName: string;
    canPublish: boolean;
    canSubscribe: boolean;
    canPublishData: boolean;  // chat via data channels
  }): Promise<VideoToken>;

  abstract getRoomParticipants(hubId: string, roomId: string): Promise<VideoParticipant[]>;

  abstract muteParticipant(hubId: string, roomId: string, identity: string, trackType: 'audio' | 'video' | 'both'): Promise<void>;

  abstract removeParticipant(hubId: string, roomId: string, identity: string): Promise<void>;

  abstract startRecording(hubId: string, roomId: string, options: RecordingOptions): Promise<string>; // egressId

  abstract stopRecording(hubId: string, roomId: string, egressId: string): Promise<void>;

  abstract testConnection(): Promise<boolean>;
}
```

### LiveKit Implementation

`src/server/video/livekit-adapter.ts` — implements VideoAdapter using:
- `livekit-server-sdk` for server-side room management and token generation
- Room names prefixed with hubId: `{hubId}:{roomName}` to prevent cross-hub collisions
- Token generation uses LiveKit's `AccessToken` with `VideoGrant` specifying room + permissions
- Recording delegates to LiveKit Egress API (see Spec 3)

### Adapter Registration

`src/server/lib/adapters.ts` — new `getVideoAdapter()` function. Currently returns LiveKitAdapter if `LIVEKIT_URL` is set, otherwise returns a `NoOpVideoAdapter` that throws "video not configured" on all calls.

## E2EE via SFrame

LiveKit's built-in E2EE uses the SFrame (Secure Frame) protocol:

1. Each participant generates a per-session encryption key on join
2. Keys are exchanged via LiveKit's key distribution protocol (encrypted via DTLS between participant and server, then forwarded)
3. Media frames are encrypted before being sent to the SFU
4. The SFU forwards encrypted frames it cannot decrypt — zero-knowledge for media content
5. Recipients decrypt using the shared session key

### Configuration

E2EE is controlled per-meeting via the `e2eeRequired` flag. When enabled:
- LiveKit room is created with E2EE enabled
- Client SDK initializes with `E2EEOptions` and a `KeyProvider`
- All participants must support E2EE or they cannot join
- Recording requires the Egress service to join as a keyed participant (see Spec 3)

When disabled (e.g., large public trainings where E2EE overhead is unacceptable):
- Media is still encrypted in transit via DTLS-SRTP
- SFU can see media for compositing/recording
- This is explicitly communicated to participants in the UI

### Client Integration

Frontend uses `@livekit/components-react` for the meeting UI:
- `LiveKitRoom` provider wraps the meeting page
- `VideoConference` component for the main view
- `ParticipantTile`, `TrackToggle`, `Chat` for controls
- E2EE initialized via `useE2EE()` hook from the SDK

## Nostr Event Types

New events published to the existing strfry relay:

```typescript
'meeting:created'           // Meeting scheduled
'meeting:started'           // Room opened, host joined
'meeting:ended'             // Room closed
'meeting:participant:joined' // Someone joined
'meeting:participant:left'   // Someone left
'meeting:recording:started'  // Recording began
'meeting:recording:stopped'  // Recording ended
```

All events hub-key encrypted, tagged with `["t", "llamenos:event"]` — same pattern as existing call/voicemail events. Relay cannot distinguish meeting events from call events.

## Scale Considerations

- Single LiveKit node supports up to ~100 concurrent participants per room
- Simulcast enabled by default (3 quality layers: high/medium/low) — bandwidth adapts per subscriber
- For rooms >50 participants, consider disabling participant video by default (audio + screen share only)
- LiveKit's built-in metrics exposed at `/metrics` for Prometheus scraping
