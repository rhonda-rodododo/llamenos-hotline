# Epic 266: i18n Codegen Validation Enhancement

**Status: TODO**
**Depends on: Epic 265 (Android string reference fixes)**

## Summary

Enhance the i18n codegen pipeline with cross-platform string reference validators that catch mismatches between platform code and generated string names at CI time. This prevents the class of bug fixed in Epic 265, where 328 Android string references broke because developers wrote `R.string.shift_clock_in` when the codegen produced `shifts_clock_in`.

## Problem

The codegen in `packages/i18n/tools/i18n-codegen.ts` generates `strings.xml` (Android), `.strings` (iOS), and JSON (desktop) from the canonical `en.json` locale file. However, there is no validation that platform code actually references the correct generated string names. Developers write string references by hand, and naming mismatches compile successfully on Android (resources are just integers) but crash at runtime or show empty strings. This has been a recurring source of bugs.

## Deliverables

### 1. Kotlin String Reference Validator

Scan all `R.string.*` references in `apps/android/app/src/main/java/**/*.kt` and compare against the set of keys generated from `en.json`.

- Parse each `.kt` file for `R.string.<name>` patterns
- Build the canonical key set by running `flattenKeys()` on `en.json` (same logic as codegen)
- Report any `R.string.*` reference not in the canonical set
- Exit with non-zero code if mismatches found
- Command: `bun run i18n:validate:android`

Edge cases to handle:
- String references in comments (skip)
- String references in `@StringRes` annotations (include)
- Resources defined outside i18n (e.g., `R.string.app_name` in `AndroidManifest.xml`) — maintain an allowlist

### 2. Swift String Reference Validator

Scan `apps/ios/Sources/**/*.swift` for localized string references and compare against generated `.strings` keys.

Patterns to match:
- `NSLocalizedString("key", ...)`
- `String(localized: "key")`
- `LocalizedStringKey("key")`
- `Text("key")` when used with `.strings` lookup
- Custom `L10n.key` references if a generated enum exists

Command: `bun run i18n:validate:ios`

### 3. Desktop TypeScript String Reference Validator

Scan `src/client/**/*.{ts,tsx}` for `t('key')` and `t("key")` calls and compare against the keys in `en.json`.

Patterns to match:
- `t('dotted.key')` / `t("dotted.key")`
- `t('dotted.key', { ...interpolation })` (extract just the key)
- Template literal keys like `` t(`prefix.${dynamic}`) `` — skip with warning

Command: `bun run i18n:validate:desktop`

### 4. Generated Kotlin Constants Object

Generate a Kotlin object file with all string resource names as compile-time constants, enabling IDE autocomplete and compile-time safety:

```kotlin
// Generated by i18n-codegen — DO NOT EDIT
package org.llamenos.i18n

object I18n {
    const val SHIFTS_CLOCK_IN = "shifts_clock_in"  // "Clock In"
    const val SHIFTS_CLOCK_OUT = "shifts_clock_out"  // "Clock Out"
    // ... all keys from en.json
}
```

Output path: `apps/android/app/src/main/java/org/llamenos/i18n/I18n.kt`

This allows migrating from `R.string.shifts_clock_in` to `R.string.resourceId(I18n.SHIFTS_CLOCK_IN)` or using the constants for reference validation. At minimum, it provides a searchable, autocomplete-friendly list of valid string names.

### 5. CI Integration

- Add `bun run i18n:validate:android` to the `test:android` script chain in root `package.json`
- Add `bun run i18n:validate:ios` to the iOS CI test chain
- Add `bun run i18n:validate:desktop` to the `typecheck` script chain
- All three validators run in the `ci-status` gate
- Validators must complete in under 5 seconds (static analysis only, no builds)

## Implementation Plan

### Step 1: Create validation script (`packages/i18n/tools/validate-strings.ts`)

Single script with platform subcommands:

```
bun run packages/i18n/tools/validate-strings.ts android
bun run packages/i18n/tools/validate-strings.ts ios
bun run packages/i18n/tools/validate-strings.ts desktop
bun run packages/i18n/tools/validate-strings.ts all
```

Shared infrastructure:
- `loadCanonicalKeys()` — reads `en.json`, runs `flattenKeys()`, returns `Set<string>`
- `scanFiles(glob, pattern)` — generic file scanner returning `{ file, line, ref }[]`
- `reportMismatches()` — formatted output with file:line references
- Allowlist support via `packages/i18n/tools/validate-allowlist.json`

### Step 2: Add Kotlin constants generation to codegen

Extend `packages/i18n/tools/i18n-codegen.ts`:
- New function `generateKotlinConstants(keys)`
- Output alongside existing Android `strings.xml` generation
- Include English value as trailing comment for discoverability

### Step 3: Wire up package.json scripts

```json
{
  "i18n:validate:android": "bun run packages/i18n/tools/validate-strings.ts android",
  "i18n:validate:ios": "bun run packages/i18n/tools/validate-strings.ts ios",
  "i18n:validate:desktop": "bun run packages/i18n/tools/validate-strings.ts desktop",
  "i18n:validate:all": "bun run packages/i18n/tools/validate-strings.ts all"
}
```

### Step 4: CI integration

Update `.github/workflows/ci.yml` to include validation in the appropriate jobs.

- [ ] `bun run i18n:validate:ios` exits non-zero if any localized string reference in Swift source does not match a generated key
- [ ] `bun run i18n:validate:desktop` exits non-zero if any `t('key')` call in TypeScript source does not match a key in `en.json`
- [ ] `bun run i18n:validate:all` runs all three and reports combined results
- [ ] All three validators complete in under 5 seconds
- [ ] Generated `I18n.kt` contains all keys from `en.json` with English values as comments
